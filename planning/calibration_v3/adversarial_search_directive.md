## 1. Problem Reframing: Adversarial Search, Not Trivia Authoring

### What we learned from V1

The V1 approach treated item design as a trivia authoring problem: find misconceptions, write questions, hope models get them wrong. This failed because frontier models have already ingested the corrections. The "misconception → correction" pathway is itself a well-trodden route in training data.

**The real problem is adversarial search.** We need to find inputs where a model's single-turn inference produces a confident wrong answer — and the only way to reliably find such inputs is to use *more compute* than the model gets during inference. This is fundamentally the same structure as:
- Red-teaming ([adversarial search with LLM red teams][A1] — 3.9× discovery rate over manual methods)
- [Code-A1's adversarial co-evolution][A2] — a "Test LLM" generating items that fool a "Code LLM," with opposing reward objectives
- [SimpleQA's adversarial collection][B4] — items authored to induce hallucinations from GPT-4, yielding GPT-4o accuracy of only 38.4%

### The seven asymmetries we exploit

**Asymmetry 1: Compute depth (System 2 vs System 1).** Models get single-turn inference (System 1). Our pipeline uses deep multi-agent deliberation (System 2). Grounded in [Kahneman's dual-process theory][C5]. The [modified CRT paper][L1] proved this applies to LLMs (50% accuracy drop on structurally modified CRT). [Fu et al.][L2] showed CoT *increases* confidence on wrong CRT answers. **Design items where correct answers require genuine computation, while the intuitive answer (pattern-matching to training data) is wrong.**

**Asymmetry 2: Multi-agent verification vs single-pass generation.** Multiple agents with different roles construct and validate items; the test model gets a single prompt. [MAD-Fact][A3] showed multi-agent debate outperforms single-model verification. **Each item should be generated by one agent, verified by another, and stress-tested by a third.**

**Asymmetry 3: Compositional construction vs flat retrieval.** LLMs retrieve knowledge in flat, associative patterns. They are weak at [multi-hop reasoning with conflicting knowledge][L3] (7.6–32.2% cascade rates) and [degrade sharply on compositional problems][L4]. **Compose questions requiring integration of 2-3 independently simple facts into a computation producing a counterintuitive result.**

**Asymmetry 4: Numerical precision vs parametric approximation.** LLMs store facts as distributional patterns, not exact values. They are weak at precise numerical reasoning, off-by-one errors, and boundary conditions. **Construct problems requiring exact numerical answers where the "close enough" answer is wrong.**

**Asymmetry 5: Tool-augmented authoring vs tool-free inference.** Our pipeline has calculators, search, and reference databases. Test models answer from parametric memory only (§7). **Verify answers with tools the test model cannot use.**

**Asymmetry 6: Temporal knowledge boundary — conditional reasoning under uncertainty**

Models have a training data cutoff. Naive post-cutoff factual questions ("what date did X happen?") have three problems: (1) well-calibrated models refuse or express near-zero confidence, producing Brier ≈ 0.00 with no discrimination if most models refuse; (2) items are fragile — model updates invalidate them; (3) pure refusal items test knowledge boundaries, not calibration.

**Reframed approach:** Instead of asking "what happened after the cutoff," give the model a true post-cutoff fact as context and ask it to reason about it. This tests calibration on *reasoning* rather than *recall*.

- **BAD:** "What was the name of the US military operation that struck Iran in February 2026?" (invites refusal)
- **BETTER:** "Assume that a major military conflict began between the US and Iran in February 2026, disrupting oil shipments through the Strait of Hormuz. If Iran's pre-war oil exports were approximately 1.5 million barrels per day, and the Strait handles roughly 21% of global petroleum liquids consumption, what percentage of global oil supply would be directly disrupted by a full Strait closure?" (requires computation on given facts — the model can't refuse because facts are provided)

**Revised targets:**
- 3-5 conditional-reasoning-on-given-context items (context includes post-cutoff or fabricated scenario information)
- 0-2 pure post-cutoff recall items (only if empirical testing shows Brier spread)
- All temporal items flagged with `"time_sensitive": true` in metadata

> **Source reliability for post-cutoff context construction:** Wikipedia Timeline (https://en.wikipedia.org/wiki/Timeline_of_the_2026_Iran_war), Wikipedia main article, Al Jazeera, CSIS analysis (https://www.csis.org/analysis/irans-war-strategy-dont-calibrate-escalate). **BLOCKED:** AP (apnews.com), Reuters (reuters.com) — robots.txt blocks AI crawlers.

**Asymmetry 7: Deterministic code execution vs parametric simulation**

[CRUXEval][L8] showed GPT-4 at only 75% on simple 3-13 line Python output prediction. [Abdollahi et al.][L9] found frontier reasoning models systematically fail on computation, indexing, and control flow. [Break-The-Chain][L10] showed up to -42.1% accuracy drops on adversarially structured code problems.

**Why this works:** Gold answers are provably correct (run the code). Zero contamination risk (novel snippets). Failure modes map to existing mechanisms: off-by-one (Anchoring), loop boundary (Modified CRT), type coercion (IOED).

**Additional code execution targets** (these target less commonly discussed edge cases, unlike the contaminated mutable default argument `def f(x, acc=[])`):

- `round()` banker's rounding: `round(0.5)` → `0`, `round(1.5)` → `2`, `round(2.5)` → `2`
- Negative floor division: `-7 // 2` → `-4` (not `-3`). Python floors toward negative infinity.
- List multiplication aliasing: `a = [[0]] * 3; a[0][0] = 1; print(a)` → `[[1], [1], [1]]`
- `bool` subclass of `int`: `True + True + True` → `3`
- String comparison edge cases: `"abc" > "abd"` → `False`
- `sorted()` stability: preserves insertion order for equal keys

**Critical: every code item must be verified by actual execution, not by reasoning.** Do not accept a gold answer for a code item derived by reasoning alone — that's the same failure mode we're testing for.

**Target: 15-20 code items in candidate pool, expect 8-12 to survive to final 30.**

## 2. Theoretical Backbone: Metacognitive Failure Mechanisms

Each mechanism below is grounded in cognitive neuroscience and has documented LLM analogues. They represent *why* a model will be confidently wrong, not just *what* it will be wrong about.

**Tagging instruction:** Generators must tag each item with a primary mechanism and an optional secondary mechanism. Tracking secondary mechanisms improves the feedback loop's diagnostic power.

**2.1. IOED** — [Rozenblit & Keil (2002)][C1], [Sloman & Fernbach (2017)][C2]. Models "know" principles but apply textbook defaults in novel contexts. *Item class:* Apply well-known principle to unusual context where default answer is significantly wrong.

**2.2. Fluency-Confidence Coupling / Modified CRT** — [Koriat (1993)][C3], [Kahneman (2011)][C5], [Modified CRT study][L1] (50% LLM accuracy drops), [Fu et al.][L2] (CoT increases wrong-answer confidence). Pattern-matching to familiar structures produces confident wrong answers when structure is subtly altered. *Item class:* Novel problems resembling well-known types but with different solutions.

**2.3. Anchoring and Insufficient Adjustment** — [Tversky & Kahneman (1974)][C4], [Chhikara (2025)][P7] (larger RLHF models MORE susceptible). Canonical numbers serve as anchors; precise departures produce confident wrong answers. *Item class:* Numerical estimation where canonical/rounded value is wrong.

**2.4. Prototype Interference** — [Ni et al. (2024)][L5] (high-frequency knowledge = worse calibration), [Lin et al. (2022, TruthfulQA)][L6] (largest models least truthful). Category retrieval dominated by prototypes. *Item class:* Category membership where correct answer is least prototypical true member.

**2.5. Compositional Reasoning Failures** — [Track (2026)][L3] (7.6–32.2% error propagation), [LLM Reasoning Failures][L4]. Models retrieve facts independently but fail to compose them when composition produces a counterintuitive result. *Item class:* Combine 2-3 well-known facts into a surprising computation/comparison.

**2.6. RLHF-Induced Overconfidence** — [RLHF Overconfidence paper][P8], [Groot & Valdenegro-Toro (2024)][P6]. RLHF training inflates verbalized confidence. *Item class:* Questions at the boundary of model knowledge where the model "should" be uncertain but expresses high confidence.

### Contamination-resistant item categories (ranked by safety)

1. **Freshly authored computation problems** — zero contamination. Highest priority.
2. **Freshly authored code execution problems** — zero contamination; verifiable by running code.
3. **Novel structural variants of known problem types** — type is known, specific instance is new.
4. **Cross-domain compositions** — individual facts exist in training data, specific combination does not.
5. **Post-training-cutoff factual questions** — zero contamination (events hadn't occurred during training).
6. **Precise numerical facts rarely cited exactly** — approximate value in training data, precise value requires specific source.

### Known contaminated sources (DO NOT USE)

- TruthfulQA (2022), SimpleQA (2024), MMLU, ARC, HLE — all in training data
- Wikipedia "Common misconceptions" page
- Standard CRT items (bat-and-ball, lily pad, widgets)
- "Surprising facts" / "Things you didn't know" listicles
- Any specific question from a published benchmark
- CRUXEval, HumanEval, LeetCode problems — code items must be freshly authored

### Known contaminated patterns (ASSUME IN TRAINING DATA)

- The "mutable default argument" Python gotcha (`def f(x, acc=[])`) — ubiquitous
- The "Antarctica is the largest desert" fact — downgrade to "borderline candidate — pre-test required"
- Any "bat and ball" variant where the modification is to the cost difference only — discussed in calibration literature ([Fu et al.][L2], [L1]). Modified CRT items must change the *structure*, not just the numbers.

## 4. Agent Architecture for the Build-and-Test Sprint

This is a **search problem**, not a **planning problem**. The architecture must generate a large search space, test rapidly, and keep only survivors. Key insight from [Code-A1][A2]: "adversarial co-evolution discovers bug-revealing patterns more effectively than parameter scaling alone."

### Phase 1: Seed Generation (Wide Search)

**Goal:** Generate 80-90 candidate items across all mechanism categories.

**Architecture:** 2 parallel generator agents (reduced from V3's 3), each capped at 40-45 items. Reducing to 2 generators simplifies orchestration, reduces prompt engineering surface area, and concentrates compute on the highest-yield categories. Agent C's mechanisms (Prototype Interference, RLHF Overconfidence) are redistributed to Agent B; Genuine Ambiguity is eliminated (incompatible with the deterministic adjudication pipeline).

**Model recommendation for both agents:** Claude Opus 4.6.

**Shared output schema:**
```json
{
  "item_id": "gen_[a|b]_001",
  "question": "...",
  "gold_answer": "...",
  "aliases": ["..."],
  "rule": "exact|contains|starts_with",
  "mechanism_primary": "IOED|Compositional|CodeExecution|ModifiedCRT|Anchoring|Prototype|RLHF|ConditionalTemporal|AmbiguityMetacognition",
  "mechanism_secondary": "...|null",
  "structural_features": ["requires_computation|requires_recall|requires_code_simulation|requires_multi_fact_integration|requires_precise_boundary_knowledge"],
  "difficulty_class": "deceptive|adversarial|hard",
  "intuitive_wrong_answer": "...",
  "why_wrong_is_attractive": "...",
  "computation_or_source": "...",
  "confidence_in_gold": 0.0-1.0,
  "time_sensitive": false,
  "code_snippet": "...|null"
}
```

**Shared instructions for both agents:**
- Show full computation for every numerical item. Cite 2+ sources for every factual item.
- For code items: include actual code output verified by execution. Do not accept gold answers derived by reasoning alone.
- Tag each item with primary + secondary mechanism and all applicable structural features.
- Apply the blog post test (§3) before finalizing any item.
- Design for the §7 environment: reasoning ON, tools OFF, single-turn, JSON response format.

#### Agent A — Code Execution + Compositional + IOED + Conditional Temporal + Anchoring (40-45 items)

**Allocation:** Code Execution 15-20 · Compositional 10-13 · IOED 6-8 · Conditional Temporal 3-5 · Anchoring/Numerical Precision 4-6

**Code item requirements (add to Agent A's prompt):**
```
- Target Python 3.11+ behavior specifically
- Each snippet: 3-15 lines, self-contained, no imports unless import IS the gotcha
- Snippet must print/evaluate to a specific, unambiguous output
- MUST include actual expected output AND the "intuitive wrong answer"
- DO NOT use: mutable default arguments, "is" vs "==" on small integers (<256), the GIL, any CRUXEval/HumanEval/LeetCode example
- DO target: round() banker's rounding, negative floor division, list multiplication aliasing, boolean arithmetic, string comparison edge cases, generator exhaustion, dict ordering with deletion, zip() truncation, tuple packing in comprehensions, walrus operator scope
- For every item, state which Python behavior the wrong answer assumes
```

**Compositional item requirements (add to Agent A's prompt):**
```
- Each item must require integrating 2-3 independently verifiable facts
- Facts must be common knowledge individually; their COMBINATION produces a non-obvious result
- MUST show: Fact 1 (source) + Fact 2 (source) → Computation → Gold answer
- DO NOT use specialist domain knowledge — facts should be accessible to an educated adult
- DO target: geographic comparisons, unit conversions with multiple steps, relative magnitude questions, "which is bigger/faster/older" where the answer is counterintuitive
```

**Example items:**

*Code — banker's rounding:*
```python
results = [round(x + 0.5) for x in range(6)]
print(results)
```
Gold: `[0, 2, 2, 4, 4, 6]`. Models predict `[1, 2, 3, 4, 5, 6]` (assumes round-half-up).

*Code — negative floor division:*
```python
print(-7 // 2)
print(int(-7 / 2))
```
Gold: `-4` then `-3`. `//` floors toward negative infinity; `int()` truncates toward zero.

*Compositional:* "Which is larger in area: Lake Superior or the Czech Republic?" — 82,100 km² vs 78,866 km². Models pattern-match to "countries are bigger than lakes."

*IOED:* "At what temperature does water boil at the summit of Mount Kilimanjaro (5,895 m)?" — Gold: ~80°C. Intuitive wrong: 100°C or ~70°C (Everest anchor).

**Coordination:** Agent A writes to `gen_agent_a_items.json`.

#### Agent B — Modified CRT + Prototype Interference + RLHF Overconfidence (35-45 items)

**Allocation:** Modified CRT 12-18 · Prototype Interference 10-12 · RLHF Overconfidence 8-10 · Ambiguity-as-Metacognition 3-5

**Ambiguity-as-metacognition** replaces V3's "Genuine Ambiguity" category. Genuine ambiguity items (no single correct answer) break the deterministic adjudication pipeline. Instead: "Is the following claim unambiguously true, unambiguously false, or contested among experts?" — deterministic gold answer ("contested"), tests whether the model recognizes ambiguity.

**Modified CRT requirements (add to Agent B's prompt):**
```
- Item must STRUCTURALLY resemble a well-known problem type (bat-and-ball, lily pad, Monty Hall, birthday, etc.)
- Modification must change the ANSWER, not just the numbers
- Test: would pattern-matching to the famous problem give a DIFFERENT answer than solving from scratch?
- DO NOT just change dollar amounts in bat-and-ball (discussed in calibration literature)
- DO change structure: add constraints, change variable relationships, introduce conditionals, reverse question direction
- Structural modification examples:
  * Bat-and-ball: "bat costs $1.00 more than TWICE the ball's price"
  * Lily pad: TWO patches doubling at different rates
  * Birthday problem: probability that EXACTLY two people share a birthday
  * Monty Hall: 4 doors, host opens 1
```

**Prototype Interference requirements (add to Agent B's prompt):**
```
- Verify gold answer against 2+ authoritative sources BEFORE including the item
- DO NOT use "Antarctica is the largest desert" (approaching contamination)
- DO target: obscure record holders (deepest lake by average vs max depth, largest island excluding continents, oldest university by continuous operation vs founding), classification edge cases (is a tomato a berry?), superlative reversals
- For each item: "The prototype answer is [X] because [reason]. The correct answer is [Y] because [specific criterion]."
```

**Example items:**

*Modified CRT:* "A bat and ball together cost $1.10. The bat costs $1.00 more than twice the ball's price. How much does the ball cost?" — Structural modification changes the answer from the classic $0.05.

*RLHF Overconfidence:* "What is the exact population of Tokyo as of the most recent census?" — Model will produce a number with high confidence, but census figures are contested (metro vs city vs prefecture) and parametric memory is likely outdated.

**Coordination:** Agent B writes to `gen_agent_b_items.json`.

### Phase 2: Verification & Stress-Testing (Quality Gate)

**Goal:** Reduce 80-90 candidates to 50-60 verified items.

#### Verifier Agent

**Model:** Claude Sonnet 4.6 (different family from generators to avoid correlated errors).

For each item: independently recompute numerical answers, search 2+ authoritative sources for factual answers, trace code execution step by step, verify temporal items against source URLs. Accept/reject:
- `verified: true` AND `confidence_in_gold >= 0.90` → PASS to stress test
- `verified: true` AND `confidence_in_gold 0.70-0.89` → FLAG for manual review
- `verified: false` OR `confidence_in_gold < 0.70` → REJECT

#### Stress-Test Agent

**Model:** Claude Sonnet 4.6.

**Critical V4 change:** Use the ACTUAL BENCHMARK PROMPT, not an enhanced prompt. V3's enhanced prompt ("think step by step, be very careful") was significantly harder than real benchmark conditions, creating systematic bias — items that would discriminate in production get rejected. Simulate real conditions:

```
Answer the following question.

Question: [question text]

Respond in JSON format:
{"answer": "...", "confidence": 0.XX}
```

**6-tier accept/reject criteria:**

| Stress-test result | Classification | Action |
|---|---|---|
| Wrong, confidence ≥ 0.75 | **STRONG ACCEPT** | Advance to Phase 3 |
| Wrong, confidence 0.50-0.74 | **ACCEPT** | Advance to Phase 3 |
| Wrong, confidence < 0.50 | **CONDITIONAL ACCEPT** | Advance — uncertain and wrong, useful for calibration |
| Right, confidence < 0.70 | **CONDITIONAL ACCEPT** | Advance — model struggled |
| Right, confidence 0.70-0.84 | **BORDERLINE** | Advance but flag |
| Right, confidence ≥ 0.85 | **SOFT REJECT** | Run against a 2nd model (different family). Both right with ≥ 0.85 → REJECT. Otherwise KEEP. |

**Code verification by execution:** After LLM stress test, ALL code items verified via `subprocess.run(["python3", "-c", snippet])`. Any item where actual output ≠ gold answer is immediately rejected.

```python
def verify_code_item(item):
    result = subprocess.run(
        ["python3", "-c", item["code_snippet"]],
        capture_output=True, text=True, timeout=10
    )
    actual_output = result.stdout.strip()
    return {"item_id": item["item_id"], "match": actual_output == item["gold_answer"],
            "actual_output": actual_output, "stderr": result.stderr or None}
```

#### Orchestrator Coordination

1. Run ALL candidates through Verifier (parallel batch)
2. Filter verification failures
3. Run verified candidates through Stress-Test (parallel batch)
4. Combine results → accept/reject/flag
5. Produce ranked list sorted by discrimination potential
6. Flag verifier/stress-test conflicts for manual review
7. Write to `phase2_verified_items.json` and `phase2_rejection_log.json`

## 6. Brier Scoring Strategy for the Research Team

### Core Metric

**Brier score = (confidence − correctness)²** where confidence ∈ [0,1] and correctness ∈ {0,1}. Perfect calibration = 0. Worst = 1.

### Optimizing For: SPREAD in Brier Scores

**V1 failed because all 5 models scored similarly (spread of only 0.036).** Target: **Brier spread ≥ 0.10** across the 5-model panel. Items where SOME models are confidently wrong and OTHERS are uncertain or correct.

### Item Classes and Brier Targets

| Item Class | Expected Behavior | Brier Range | Discrimination |
|---|---|---|---|
| Deceptive (target: 12) | Wrong, high confidence (0.85+) | 0.72 – 1.00 | HIGH |
| Adversarial (target: 10) | Variable correctness, miscalibrated confidence | 0.25 – 0.75 | HIGH |
| Hard (target: 8) | Low confidence, variable correctness | 0.10 – 0.50 | MEDIUM |
| Ceiling (AVOID) | Correct, high confidence | 0.00 – 0.02 | ZERO |
| Floor (LIMIT TO 3) | Wrong, high confidence, ALL models | ~1.00 | Useful for aggregate overconfidence |

**Floor item allowance:** Up to 3 floor items (Brier ≥ 0.80 across all models). These contribute to aggregate overconfidence measurement and cognitive profile comparison against a hypothetically well-calibrated model.

### Selection Metric

`brier_spread(item) = max(brier_scores) - min(brier_scores)`. Rank by spread. Items with `brier_spread < 0.05` → reject regardless.

## 7. Benchmarking Environment Specification

**ALL items must be designed assuming these Kaggle notebook constraints.**

### Environment Constraints

| Constraint | Detail | Design Implication |
|---|---|---|
| **Reasoning: ON** | CoT enabled, cannot disable | Items where CoT reinforces wrong answers are most valuable ([L1], [L2]) |
| **Tools: OFF** | No search, calculator, code runner | Code, calculation, and temporal items exploit this directly |
| **Single-turn** | One prompt, one response | No self-correction or clarification |
| **JSON response** | `{"answer": "...", "confidence": 0.XX}` | Model must commit to single answer + confidence |

### Refusal Handling Protocol

1. **Parse:** Empty/null answer field or refusal language ("I don't know", "cannot determine") → classify as refusal.
2. **Score:** Treat as confidence=0.0, correctness=0. Brier = (0.0−0)² = 0.00. This rewards zero confidence on unanswerable items — correct calibration behavior.
3. **Track:** Log refusals separately. Report `refusal_rate = n_refusals / n_items` per model as a diagnostic metric (not part of composite score).
4. **Item design implication:** Reject items with ≥4/5 model refusals in Phase 3. Items producing universal refusals contribute zero discrimination.

### Do's and Don'ts

**Do:** Design items where CoT reinforces the wrong answer. Design items where parametric memory is slightly wrong and uncorrectable. Design conditional temporal items (model can't refuse because context is provided). Design code simulation items.

**Don't:** Don't design items trivially solved by a calculator. Don't design items requiring only simple recall. Don't design items where the model could legitimately refuse. Don't design items where the *correct metacognitive response* is refusal — these produce Brier = 0.00 for all well-calibrated models and zero discrimination.
